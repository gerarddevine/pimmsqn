General Introduction

The CMIP5 metadata questionnaire will provide vital information to help users of the CMIP5 database understand why different models respond in unique ways to the forcings defined by the CMIP5 experiment protocol. Completion of the questionnaire is a necessary step towards the publication of your data. The value of the metadata captured by the questionnaire is down to you. The questionnaire will ask you to describe core information about your models and it also gives you the freedom to share further model characteristics as you see fit. Once all the core information has been described you will be able to submit your questionnaire entries to CMIP5. Completing the CMIP5 questionnaire is a demanding task, support will be provided to you by the Metafor team who designed and developed the questionnaire, under the guidance of the CMIP panel, with individuals from the BADC, University of Reading, MetOffice and CERFACS.

Basic Concepts

  1. Each CMIP5 modelling centre is performing Simulations which run Models on Platforms.
  2. The Models are made up of Components.
  3. Simulations Conform to the Numerical Requirements of Experiments.
  4. Conformances consist of either specific code modifications or the use of specific boundary or initial condition Files.
  5. The top level components under models (also known as Realms) can be Coupled either to other components, or to files.
    • Internal Couplings describe couplings to other components.
    • External Couplings describe couplings to files.

Purpose

The purpose of this questionnaire is to glean information about...

  • Models, Components and Platforms,
  • Simulations, Experiments and Numerical Requirements,
  • Conformances, Files and Coupling,
and/or their relationships.

Typical Path Through the Questionnaire

A roadmap to guide you along the typical path through the quesitonnaire has been produced for you to download either as a dynamic powerpoint document or as a static pdf file.

We expect most folk will begin by describing a model. You will need to describe the model in enough detail to a) validate the model using the validate button (which will tell you which sections remain incomplete), and b) have enough inputs described to be able to answer conformance questions in the simulations (most of these are about how the model met various boundary conditions, so you will need the input requirements for those boundary conditions to be described. However, validation is not necessary before beginning to fill out a simulation (although a platform description is necessary), and one can go through an iterative process of trying to fill out the experiment conformances and going back to the model to describe inputs, then filling out boundary conditions (either as defaults in the model, or in the simulation descriptions, or both).

As indicated above, you will need a platform description before filling out a simulation, but that wont be hard ...

When describing simulations you will be asked to clearly explain how the simulation has met the numerical requirements of the experiment with which it is associated. This is done via the conformance page, which maps the numerical requirements onto initial and boundary conditions (and, if necessary, onto code modifications) . You will probably have to dive off and create some initial conditions.

If you are creating an ensemble, you'll need to describe the ensemble members. We have the concept of code modifications for physical ensemble members, and also for code modifications to a basic state model needed to conform to a numerical experiment. You can create and assign those from either the ensemble interface or the conformance interface.

Throughout you may need to add references, and make links between components or simulations, and those references.
